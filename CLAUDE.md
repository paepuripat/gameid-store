## Project Overview

**gameid-store** вЂ” a Thai digital-product store that sells game account IDs. Customer picks a product, pays via a PromptPay QR generated for the order amount, uploads the bank slip, and the system verifies the slip with **SlipOK** (real bank-slip verification: amount cross-check + duplicate detection). On success it claims one credential from D1 inventory and reveals the game ID (username / password / notes) on screen.

Goal: prove the **real slip-verification + on-screen delivery** loop, deployed live. The app ships to a live Cloudflare Workers URL by the end of EP.1.

## Non-Negotiable Scope Rules

These are intentional design constraints. **Do not expand scope without being asked.**

- вњ… Tier 3 stack **minus auth**: SPA + Hono API Worker + D1 + Drizzle. **No Better Auth in EP.1.**
- вњ… Web only (responsive). No mobile app.
- вњ… No email sending in EP.1 вЂ” delivery is on-screen reveal. (Email is EP.2.)
- вњ… No admin UI in EP.1 вЂ” products and inventory are seeded into D1 via `seed.sql`. (Admin is EP.3.)
- вњ… Single-item purchase only. No cart, no customer accounts, no profiles.
- вњ… Do not store the uploaded slip image. Forward it to SlipOK and discard.
- вњ… The SlipOK API key and Branch ID live ONLY in the Worker `env`, never in client code or a `VITE_`-prefixed var.
- вњ… The PromptPay ID is config (`VITE_PROMPTPAY_ID`) and may live client-side вЂ” it is encoded in the QR anyway, so it is not a secret.

If a feature is not listed in the checkpoints below, **ask before adding**.

## Tech Stack (Locked)

Verified current at build time (June 2026):

- **Cloudflare Workers** with static assets вЂ” deploy target. NOT Pages.
- **Vite + React + TypeScript**
- **`@cloudflare/vite-plugin`** (v1.40.x) + **wrangler** вЂ” dev server runs the Worker in workerd; one origin, no CORS. `vite.config.ts` plugins: `[react(), cloudflare()]`.

- **qrcode.react** вЂ” render the payload with the named export `QRCodeSVG`.
- **react-router-dom** (v7.x) вЂ” client-side routing for the SPA (Storefront в†’ ProductDetail в†’ Checkout в†’ Success).
- **SlipOK API** вЂ” bank-slip verification. `POST https://api.slipok.com/api/line/apikey/<BRANCH_ID>`, header `x-authorization: <API_KEY>`.
- **Package manager:** pnpm
- **No** Better Auth, no admin UI, no email, no Pages, no Next.js, no Supabase.

## Type Contract (Source of Truth)

Define in `src/types.ts`, reuse in the Worker:

```typescript
export interface Product {
  id: string;
  name: string;
  description: string | null;
  imageUrl: string | null;
  price: number;       // THB, e.g. 199 вЂ” used for the QR amount AND the SlipOK cross-check
  active: number;      // 1 | 0
}

export interface CreatedOrder {
  orderId: string;
  amount: number;      // THB
  productId: string;
}

export interface RevealedCredential {
  username: string;
  password: string;
  notes: string | null;
}

// Discriminated result from POST /api/verify-slip
export type VerifyResult =
  | { ok: true; credential: RevealedCredential }
  | { ok: false; reason: "invalid_slip" | "wrong_amount" | "wrong_receiver" | "duplicate_slip" | "sold_out"; message: string };
```

## Database Schema

Drizzle schema in `worker/db/schema.ts` (SQLite / D1).

```typescript
import { sqliteTable, text, integer, real } from "drizzle-orm/sqlite-core";

export const products = sqliteTable("products", {
  id: text("id").primaryKey(),
  name: text("name").notNull(),
  description: text("description"),
  imageUrl: text("image_url"),
  price: real("price").notNull(),          // THB
  active: integer("active").notNull().default(1),
});

export const inventory = sqliteTable("inventory", {
  id: text("id").primaryKey(),
  productId: text("product_id").notNull().references(() => products.id),
  username: text("username").notNull(),
  password: text("password").notNull(),
  notes: text("notes"),
  status: text("status").notNull().default("available"), // 'available' | 'sold'
  orderId: text("order_id"),
});

export const orders = sqliteTable("orders", {
  id: text("id").primaryKey(),
  productId: text("product_id").notNull().references(() => products.id),
  amount: real("amount").notNull(),         // THB
  email: text("email"),                     // collected now, used for sending in EP.2
  status: text("status").notNull().default("pending"), // 'pending' | 'paid'
  slipTransRef: text("slip_trans_ref").unique(),        // dedup backstop; SlipOK 1012 is the primary guard
  createdAt: integer("created_at").notNull(),
});
```

Apply migrations (NOT `drizzle-kit migrate`):

```bash
wrangler d1 migrations apply gameid-store-db --remote   # before the prod demo
```

### Checkpoint 2 вЂ” Checkout + PromptPay QR

**Deliverable:**
- `POST /api/orders` accepts `{ productId }`, looks up the price **server-side** (never trust a client-sent amount), inserts a `pending` order, returns `CreatedOrder`.
- Checkout page: shows amount + email input + the PromptPay QR for the order amount + a slip upload dropzone (upload not wired to verify yet).

**Critical implementation details:**
```tsx
import generatePayload from "promptpay-qr";        // default export
import { QRCodeSVG } from "qrcode.react";           // named export (not default)

const payload = generatePayload(import.meta.env.VITE_PROMPTPAY_ID, { amount: order.amount });
// <QRCodeSVG value={payload} size={256} />
```

**Known gotchas to handle:**
- `amount` must be a number in THB, not a string, or the QR encodes no amount.
- The amount source of truth is the server (the order row); the QR just mirrors it.

**Verify:**
- Click buy в†’ checkout shows correct amount.
- Scan the QR with a real banking app в†’ correct payee + amount appears.

### Checkpoint 3 вЂ” Slip verification endpoint (proxy to SlipOK)

**Deliverable:**
- `POST /api/verify-slip` accepts multipart `{ orderId, slip(File) }`, loads the order's expected amount from D1, forwards to SlipOK with the slip file + amount cross-check + dedup logging.

**Critical implementation details:**
```ts
const form = await c.req.formData();
const slip = form.get("slip") as File;
// ...load order from D1, get order.amount...

const slipForm = new FormData();
slipForm.append("files", slip);                  // field name MUST be "files"
slipForm.append("amount", String(order.amount)); // SlipOK cross-checks the amount
slipForm.append("log", "true");                  // enables duplicate detection (error 1012)

const res = await fetch(
  `https://api.slipok.com/api/line/apikey/${c.env.SLIPOK_BRANCH_ID}`,
  { method: "POST", headers: { "x-authorization": c.env.SLIPOK_API_KEY }, body: slipForm },
);
const result = await res.json();
// Branch on result.success / result.data.success; map SlipOK error codes:
//   1012 -> duplicate_slip, 1013 -> wrong_amount, 1014 -> wrong_receiver
// On success, capture result.data.transRef for the dedup backstop.
```

**Known gotchas to handle:**
- Do NOT set `Content-Type` manually when sending `FormData` вЂ” let the runtime set the multipart boundary.
- SlipOK returns its error codes in the JSON body; read the body even on non-2xx.

**Verify:**
- With secrets in `.dev.vars`, upload a real correct slip в†’ response has `success: true` + `data.transRef`.
- Wrong amount в†’ 1013; reused slip в†’ 1012/1014 mapped to friendly reasons.

### Checkpoint 4 вЂ” Claim credential + reveal

**Deliverable:**
- On verified success, in one atomic step: mark order `paid` + store `slipTransRef`, and claim one available credential for the product. Return `RevealedCredential`.
- Success page shows username / password / notes.
- Handle `sold_out` (no available inventory) and `duplicate_slip` (transRef already used) without ever revealing a credential.

**Critical implementation details:**
- Claim atomically so two concurrent buyers can't get the same credential. Use a single guarded statement:
```sql
UPDATE inventory
SET status = 'sold', order_id = ?
WHERE id = (
  SELECT id FROM inventory
  WHERE product_id = ? AND status = 'available'
  LIMIT 1
)
RETURNING username, password, notes;
```
- Only claim AFTER confirming the slip is valid AND the amount matched. Never reveal on an unverified slip.

**Verify:**
- Real correct slip в†’ success screen reveals the ID; in D1 that inventory row is `sold` and the order is `paid`.
- Reuse the same slip в†’ "аёЄаёҐаёґаё›аё™аёµа№‰аё–аё№аёЃа№ѓаёЉа№‰а№„аё›а№ЃаёҐа№‰аё§", no new credential released.
- Empty the product's stock в†’ buying returns `sold_out`.

## Coding Conventions

- **TypeScript strict.** Types in `src/types.ts`. No `any`.
- **Hono for routing**, all API under `/api/*`.
- **Drizzle client per request** from `c.env.DB` вЂ” do not create a module-level singleton.
- **Secrets only in the Worker `env`.** Never reference the SlipOK key from client code.
- **Error handling:** every SlipOK / D1 call checks for errors and surfaces a typed reason.
- **Tailwind** only. No CSS-in-JS.
- **Commits:** one per checkpoint, `feat(checkpoint-N): {summary}`.

## Anti-Patterns to Avoid

Generic:
- вќЊ Don't add Zustand/Redux/Context unless local state genuinely won't work.
- вќЊ Don't write tests for this MVP unless asked.
- вќЊ Don't refactor finished checkpoints unless asked.

Build-specific:
- вќЊ Don't put `SLIPOK_API_KEY` / `SLIPOK_BRANCH_ID` in client code or a `VITE_` var. They live in the Worker env only.
- вќЊ Don't set `Content-Type` manually on the SlipOK `FormData` request вЂ” it breaks the multipart boundary.
- вќЊ Don't use the SlipOK field name anything other than `files`.
- вќЊ Don't trust a client-sent amount вЂ” read the order amount from D1 server-side for both the QR and the SlipOK cross-check.
- вќЊ Don't claim inventory with a select-then-update; use the single `UPDATE ... (SELECT ... LIMIT 1) RETURNING` so concurrent buyers can't get the same ID.
- вќЊ Don't reveal a credential before `result.success` (and `result.data.success`) is confirmed.
- вќЊ Don't run the Worker on a separate port and fight CORS вЂ” use `@cloudflare/vite-plugin`.
- вќЊ Don't forget `not_found_handling: "single-page-application"`, or a refresh on `/checkout` 404s.
- вќЊ Don't use `drizzle-kit migrate` against D1; generate with drizzle-kit, apply with `wrangler d1 migrations apply`.
- вќЊ Don't import `QRCode` as a default from `qrcode.react` (old API). Use the named `QRCodeSVG`.

## When Stuck

If something fails after one attempt:
1. State what you tried and the exact error.
2. Give **two** likely causes, ranked.
3. Recommend the smallest test to tell them apart.
4. Wait for direction. Don't thrash.

Common stuck-points for this build:
- `/api` 404 in dev в†’ is `cloudflare()` in `vite.config.ts` plugins?
- SlipOK 401/403 в†’ wrong header name (must be `x-authorization`) or wrong Branch ID in the URL.
- SlipOK "QR not found" в†’ slip image unclear or unsupported format (use JPG/PNG/WEBP with the QR visible).
- Deployed app 500s but local works в†’ secrets not set with `wrangler secret put`, or remote D1 not migrated/seeded.
